# EXC001 - 在 catch 中打印栈跟踪（Stack Trace in Catch）

| 属性 | 取值 |
|---|---|
| **ID** | `EXC001` / `lenovo-exc001-stack-trace-in-catch` |
| **类别** | Reliability |
| **严重级别** | Warning |
| **默认启用** | 是 |
| **对齐 C#** | [`EXC001` in AnalyzerRules.md](../../AnalyzerRules.md#exc001---invalid-stack-trace-usage) |

## 描述

`catch` 内直接调用 `backtrace_symbols` / `fprintf(stderr, ...)` 把异常信息打到标准流，会暴露内部模块结构、版本号、目录信息，存在两类风险：

1. 帮助攻击者做信息收集
2. 日志被错误聚合，丢失异常上下文（缺少关联 ID / trace ID）

应改用结构化日志（`spdlog::error("operation failed", ex)`、自家 logger 等）并保留异常上下文。

## 触发示例

```cpp
catch (const std::exception& ex) {
  void* buf[20];
  int n = backtrace(buf, 20);
  backtrace_symbols_fd(buf, n, 2);   // ❌
  fprintf(stderr, "%s", ex.what());   // ❌
}
```

## 合规示例

```cpp
catch (const std::exception& ex) {
  logger.Error("operation failed", ex);
}
```

## 已知限制

- 不区分"调试日志写 stderr"和"生产日志写 stderr"
- 通过 NOLINT 或在 `.clang-tidy` 中给 debug build 关闭
